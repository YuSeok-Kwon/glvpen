# BaseBall LOCK 2.0 - 로그인 & 인증 시스템 설계 (이메일 인증, 소셜, 자동 로그인)

---

## 주요 기능 요약

- 이메일 인증 기반 회원가입 / 비밀번호 찾기
- bcrypt 비밀번호 해시화
- 소셜 로그인 (Google, Naver, Kakao 등)
- 로그인 유지 (Refresh Token + HttpOnly 쿠키 기반)
- 보안 고려사항 정리

---

## 이메일 인증 구조

### 회원가입 시

1. 사용자가 이메일/비밀번호 입력
2. 서버에서 인증코드 생성 후 이메일 전송
3. 사용자가 인증코드 입력 → 검증 성공 시 가입 완료

### 비밀번호 찾기

1. 이메일 입력 → 인증코드 전송
2. 인증 성공 시 임시 비밀번호 발급 or 비밀번호 재설정 가능

### 보안 포인트

- 인증코드 만료 시간: 3~5분
- 하루 발송 횟수 제한 (DoS 방지)
- 인증코드는 6자리 숫자보다 문자+숫자 혼합 추천
- 인증 링크 방식도 고려 가능

---

## 비밀번호 암호화 (bcrypt)

| 항목            | 설정                                 |
| --------------- | ------------------------------------ |
| 해시 방식       | `bcrypt.hash(password, 12)`          |
| 검증 방식       | `bcrypt.compare(raw, hashed)`        |
| 권장 saltRounds | 10~12                                |
| 변경 시 재인증  | 현재 비밀번호 확인 or 메일 인증 필수 |

---

## 소셜 로그인

### 흐름

1. 프론트에서 소셜 로그인 → access_token 발급
2. 백엔드에 전달 → 사용자 정보 조회
3. 기존 계정 연동 or 자동 회원가입 처리

### 구현 포인트

- Passport.js + 각 provider 전략 사용
- 이메일 중복 시 연동 여부 확인 필요
- 최초 로그인 시 추가 정보 요청 가능 (닉네임 등)
- 소셜 연동 해제 기능도 고려

---

## 로그인 유지 기능 (자동 로그인)

### 핵심 구조

- Access Token (15분)
- Refresh Token (7~30일)
- 둘 다 **HttpOnly + Secure + SameSite 쿠키**로 저장
- Refresh Token은 DB 저장 및 매 로그인 시 갱신 (회전 방식)

### 흐름 요약

1. 로그인 성공 시 access_token + refresh_token 발급
2. 쿠키로 클라이언트에 저장
3. access 만료 시 refresh로 재발급 요청
4. refresh도 만료되면 재로그인 필요
5. 로그아웃 시 쿠키 삭제 + DB refreshToken 삭제

## 추가 고려사항

### 보안 관련

| 항목                    | 설명                                 |
| ----------------------- | ------------------------------------ |
| Secure 쿠키             | HTTPS에서만 동작                     |
| SameSite=Strict         | CSRF 방지                            |
| HttpOnly                | JS 접근 불가 (XSS 방지)              |
| 토큰에 개인정보 저장 ❌ | 이메일, 비밀번호 절대 포함하지 말 것 |
| 알고리즘 고정           | JWT `alg=HS256` 등 명시              |
| 서명키 보안             | `.env.JWT_SECRET` 최소 256bit 이상   |

### Refresh Token 보안

| 항목          | 설명                                      |
| ------------- | ----------------------------------------- |
| DB 저장       | userId + token + expiresAt                |
| 회전 방식     | 매 로그인 시 갱신 후 기존 토큰 폐기       |
| 로그아웃 처리 | DB에서 즉시 삭제                          |
| 탈취 탐지     | userAgent/IP 불일치 시 전체 로그아웃 고려 |

### 프론트엔드 처리

| 항목             | 설명                                    |
| ---------------- | --------------------------------------- |
| axios 설정       | `withCredentials: true` 필요            |
| 로그인 상태 유지 | 새로고침 시 `/auth/me`로 유저 확인      |
| 에러 대응        | 401/403 시 자동 로그아웃 or 재인증 유도 |

### refresh_tokens

| 필드      | 타입     | 설명          |
| --------- | -------- | ------------- |
| id        | int      | PK            |
| userId    | int      | FK to User    |
| token     | string   | refresh token |
| expiresAt | datetime | 만료 시각     |
| createdAt | datetime | 생성 시각     |

### account_verifications

| 필드      | 타입     | 설명                    |
| --------- | -------- | ----------------------- |
| id        | int      | PK                      |
| email     | string   | 사용자 이메일           |
| code      | string   | 인증 코드               |
| type      | string   | signup / reset_password |
| expiresAt | datetime | 만료 시각               |
| verified  | boolean  | 인증 완료 여부          |

# BaseBall LOCK 2.0 - 모드 확장 구조 설계

---

## 도입: 3가지 게임 모드

### 1. **Real Match Mode**

- 실제 KBO 스케줄 및 라인업을 기반으로 시뮬레이션
- 어제 or 당일 경기 라인업 사용 (당일은 경기 1시간 전 공개)
- 크롤링 기반 자동 저장

### 2. **Custom Hitter Mode**

- 유저가 만든 타자를 실시간 시뮬로 키우는 RPG형 시스템
- 실시간 턴 구조, 경험치/스탯 성장 포함
- 선택 기반 행동으로 성장 분기

### 3. **Classic Sim Mode**

- 기존처럼 유저 라인업 vs 봇 라인업 자동 경기
- 대신 하이라이트만 출력, 난이도(Easy/Normal/Hard) 선택 가능

---

## 전체 설계 시 고려할 주요 구조

### 1. 모드 기반 도메인 분리

- 각 모드는 컨트롤러, 서비스, DTO, 결과 구조를 독립적으로 유지
- 공통 로직은 /common 또는 /core로 분리

---

### 2. 공통 요소와 전용 요소 분리

| 공통 요소         | 모드 전용 요소                   |
| ----------------- | -------------------------------- |
| 유저/팀/선수 정보 | 성장 시스템 (customplayer)       |
| 경기 결과 저장    | 실시간 세션 (customplayer)       |
| MVP 계산          | 자동 라인업 생성 (realmatch)     |
| 경기 로그 출력    | 난이도 설정 및 봇 전략 (classic) |

---

### 3. 실시간 경기 흐름 처리 (Custom Hitter Mode)

- WebSocket 기반 턴 단위 진행
- GameSession 저장 구조 (메모리 or Redis)
- 중간 상태 유지, 재접속 처리 필요

---

### 4. 스케줄 + 라인업 크롤링 (Real Match Mode)

- 전일 라인업: 새벽 자동 크롤링
- 당일 라인업: 경기 시작 1시간 전
- 테이블 설계: schedule + lineup

---

### 5. 커스텀 타자 성장 시스템

- custom_player 테이블 + 경험치, 레벨, 능력치 컬럼
- 성장 이벤트 로그 저장
- 경기 행동에 따라 분기 성장 or 특성 부여

---

### 6. 클라이언트 구조도 모드 분리

- 각 모드별 상태/라우트/페이지 독립 구성
- 공통 컴포넌트는 props로 분기 처리

---

### 7. 공통 결과 화면 통합

- /game/result-view?scheduleId=123&mode=classic
- GameResultDTO는 공통 + 모드별 필드 조합
- UI는 mode에 따라 분기 렌더링

---

### 8. Classic Mode 난이도 시스템

- Difficulty enum: EASY / NORMAL / HARD
- 봇 라인업 구성 로직을 난이도에 따라 가중치/전략 조절
- 하이라이트 출력만 (시간 압축)

# 경기 애니메이션 및 UI 디자인 개선 제안

---

## 문제 인식: 기존 애니메이션/UI의 한계

| 항목          | 문제점                                             |
| ------------- | -------------------------------------------------- |
| 움직임        | 선수 이동이 직선적이고 딱딱함 (left/top 변화 중심) |
| 연출 다양성   | 홈런/안타/도루/아웃 간 차별화 부족                 |
| 디자인 일관성 | 카드 스타일과 경기장 UI 간 시각적 통일성 떨어짐    |
| 확장성        | 효과음, 강조 효과, 카드 확대 등 추가 연출 어려움   |

---

## 개선 방향 제안

### 1. **GSAP 애니메이션 도입**

#### 기술 스택

- [`gsap`](https://greensock.com/gsap/) (GreenSock Animation Platform)
- 자연스러운 easing, 연속 모션, 병렬 애니메이션 가능

---

### 2. **SVG 기반 경기장 UI 재설계**

| 구성 요소 | 설명                             |
| --------- | -------------------------------- |
| 베이스    | SVG 원(circle)로 구성, 위치 고정 |
| 주자/타자 | SVG 캐릭터 or 아이콘             |
| 배경      | SVG로 스타일링 가능한 필드 배경  |

→ CSS position이 아닌 **% 좌표 + transform 이동**으로 모든 위치 제어

---

### 3. **상황별 연출 디자인 예시**

| 상황     | 애니메이션/이펙트                                |
| -------- | ------------------------------------------------ |
| 안타     | 주자 이동 + 빛나는 경로 (GSAP path plugin)       |
| 도루     | 빠른 점프 이동 + 파티클 효과                     |
| 홈런     | 슬로우모션 + 불꽃 트레일 + 홈 플레이트 줌인      |
| 삼진     | 타자 흐릿해지며 아웃 표시 등장                   |
| MVP 등장 | 카드 확대 + 빛나는 반짝 효과 + 텍스트 애니메이션 |

---

### 4. **카드와 연계된 연출**

- 경기 중 특정 장면에서 선수 카드가 반짝이거나 확대됨
- 홈런 시 해당 카드가 하이라이트로 등장 (ex. Pokémon 카드 연출처럼)
- MVP 발표 시 카드가 회전하며 앞면 노출

---

## 연출 강화 구조 설계 예시

```
EventManager.ts
 ├── handlePlayResult()
 ├── → triggerAnimation(type: 'homeRun' | 'hit' | 'out')
 └── → updateCardEffect(playerId)
```

## 기술 적용 정리

| 기술           | 용도                                        |
| -------------- | ------------------------------------------- |
| GSAP           | 자연스러운 애니메이션 (이동, 확대, 효과 등) |
| SVG            | 고정 좌표 기반 경기장 및 객체 배치          |
| React + useRef | DOM 참조 기반 애니메이션 트리거             |
| CSS Modules    | 요소별 스타일 일관성 유지                   |

## 결론

> 기존 애니메이션의 한계를 극복하려면,  
> **GSAP + SVG 기반 재설계**를 통해 시각적 몰입감과 확장성을 확보하는 것이 중요하다.
