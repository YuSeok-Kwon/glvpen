# 📁 BaseBall LOCK 기준: Layer-based 구조 장단점 정리

BaseBall LOCK 프로젝트에 Layer-based 구조 (역할별 책임 분리 구조)를 적용했을 경우의 장단점을 분석한 내용입니다.

---

## 📌 Layer-based 구조란?

- 역할(책임)에 따라 폴더를 나누는 구조
- 예: controller, service, repository, model, route 등 각 계층별로 분리
- 전통적인 MVC 아키텍처에 가깝고, 많은 프레임워크에서 기본값처럼 제공

### 📁 예시 구조
```
/src
├── controllers/
│   ├── user.controller.ts
│   ├── game.controller.ts
├── services/
│   ├── user.service.ts
│   ├── game.service.ts
├── repositories/
│   ├── user.repository.ts
├── models/
│   ├── user.model.ts
├── routes/
│   └── user.route.ts
```

---

## ✅ 장점 (BaseBall LOCK 기준)

| 항목 | 설명 |
|------|------|
| **👀 역할 명확** | 컨트롤러/서비스/리포지토리 등 각 책임이 폴더로 구분돼 직관적 |
| **🧠 학습 난이도 낮음** | MVC 구조에 익숙한 개발자에게 진입장벽이 낮음 |
| **🗂 동일 레이어끼리 비교 쉬움** | 여러 도메인의 컨트롤러나 서비스 간 비교 용이 |
| **⚙️ 자동화에 유리** | 고정된 폴더 구조로 CLI, 코드 생성기 적용 쉬움 |
| **📌 리팩토링 분산 가능** | 특정 레이어만 집중적으로 리팩토링 가능 (ex. service만) |

---

## ❌ 단점 (BaseBall LOCK 기준)

| 항목 | 설명 |
|------|------|
| **📦 관련 코드 분산** | user 관련 코드가 controller/service/repo/model 등 여러 폴더에 분산 |
| **🧪 테스트 어려움** | 하나의 기능 테스트 시 여러 계층 의존성 고려 필요 |
| **🧹 유지보수 어려움** | 도메인 단위 문제 추적 시 여러 파일 오가야 함 |
| **📈 기능 확장 시 파일 분산** | 기능 하나 추가해도 여러 곳에 파일 추가/연결 필요 |
| **⛔ 도메인 경계 약화** | 비즈니스 로직이 레이어 넘나들며 결합도 증가 가능성 |

---

## 📌 BaseBall LOCK 기준 평가

| 항목 | 평가 |
|------|------|
| **복잡한 도메인 대응력** | ❌ 도메인 중심 파악이 어려움 |
| **테스트 용이성** | ❌ 단위 테스트가 번거롭고 setup 복잡 |
| **장기 확장성** | ❌ 유지보수/기능 분리 어려움 |
| **초기 MVP 구성** | ✅ 빠르게 구조 잡고 시작 가능 |

---

## 🔥 결론

> **BaseBall LOCK처럼 도메인이 복잡하고 비즈니스 로직이 풍부한 프로젝트에는 Layer-based 구조는 제한적이다.**

### 하지만 다음과 같은 상황에선 적합할 수 있다:
- 기능이 단순하거나, 도메인이 많지 않은 초기 MVP
- 팀원들이 역할 기반 구조에 익숙할 때
- 빠른 개발과 배포가 목적일 때
