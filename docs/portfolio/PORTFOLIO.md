## 1. 프로젝트 한 줄 요약

KBO 6시즌(2020~2025) 26만 건 데이터를 7개 크롤러로 수집하고, 세이버메트릭스(BABIP·FIP·wOBA·wRC+) 자동 산출 시스템과 5가지 고급 분석(클러치 지수·선수 유사도·Marcel 예측·승리 예측·피로도 분석)을 순수 Java로 구현하여, Python 배치 엔진(15개 분석 주제)과 Gemini AI 자동 기사 생성(연간 104건)까지 통합한 End-to-End 데이터 분석 플랫폼입니다.

### 이 프로젝트의 차별화 포인트

**Kick 1. "전통 지표의 함정을 깨다" — 세이버메트릭스 기반 재평가**
KBO 공식 사이트는 시즌별 기본 통계(타율·방어율)만 제공하여, 선수의 진짜 실력과 행운을 분리할 수 없었습니다. 수비 무관 투구력(FIP)과 실제 방어율(ERA)의 괴리를 정량화하여 "ERA가 낮아도 FIP가 높으면 내년 반등할 수 없다"는 예측적 인사이트를 도출했고, BABIP 행운 보정 분석으로 전통 지표에 감춰진 과대·과소 평가 선수를 식별하는 데이터 인프라를 구축했습니다.

**Kick 2. 분석에서 끝나지 않는 "스토리텔링 자동화"**
데이터 분석의 가치는 비전문가도 이해할 수 있을 때 극대화됩니다. Python 배치 엔진이 15개 주제별 통계 분석과 가설검정을 수행한 뒤, 그 결과를 Gemini AI가 "도입-분석-전망" 3단 구조의 기사로 자동 변환하는 파이프라인을 구축했습니다. 주 2회(월·목) 자동 발행으로 연간 104건의 데이터 기반 분석 컬럼을 생산하며, AI는 사전 계산된 수치만 사용하여 수치 날조를 원천 차단했습니다.

**Kick 3. 11개월, v1→v3 진화 — 개발자에서 데이터 분석가로**
v1(개발 연습)에서 시작해 v2(카드 시뮬레이션 게임 확장), v3(데이터 분석 플랫폼)으로 11개월간 프로젝트가 진화했습니다. 크롤링 → 정규화 → 파생 지표 산출 → 통계 분석 → AI 스토리텔링 → 시각화까지 전 과정을 단독으로 설계·구현하면서, "데이터로 이야기하는 것"의 가치를 체감하여 데이터 분석가로의 커리어 전환을 결심하게 되었습니다.

## 3. 문제 정의

### 비즈니스 맥락

KBO 공식 사이트와 기존 야구 통계 서비스는 **시즌별 기본 통계**(타율·홈런·방어율)만 제공할 뿐, 세이버메트릭스 기반 심층 분석이나 다시즌 횡단 비교를 지원하지 않았습니다. 타율 3할 타자와 2할 8푼 타자의 실력 차이가 정말 타율만으로 설명되는지, ERA 3점대 투수가 내년에도 그 성적을 유지할 수 있는지 — 이런 질문에 데이터로 답할 수 있는 플랫폼이 부재했습니다.

### 핵심 질문 3가지

**Q1. 전통 지표가 선수의 진짜 실력을 반영하는가?**
타율(AVG)과 방어율(ERA)은 수비·행운·구장 효과가 혼재된 결과 지표입니다. 선수의 "통제 가능한 실력"을 분리하려면 BABIP, FIP, wOBA 같은 파생 지표가 필요하지만, KBO에서는 이를 체계적으로 산출·비교할 수 있는 인프라가 없었습니다.

**Q2. 6시즌 횡단 비교로 어떤 패턴을 발견할 수 있는가?**
단일 시즌 스냅샷으로는 보이지 않는 선수의 성장·노화 곡선, 카테고리별 구조적 패턴, 정규시즌 vs 포스트시즌 성과 괴리를 식별하려면 다시즌 데이터가 정규화된 형태로 필요했습니다.

**Q3. 분석 결과를 비전문가도 소비할 수 있는 형태로 전달할 수 있는가?**
수치와 차트만으로는 인사이트 전달에 한계가 있습니다. 데이터 분석 결과를 "기사" 형태로 자동 변환하여, 야구 팬이라면 누구나 이해할 수 있는 스토리텔링이 필요했습니다.

### 왜 이 문제를 정의했는가?

야구는 "기록의 스포츠"라 불리지만, 한국 야구 팬이 접근할 수 있는 분석 도구는 미국(FanGraphs, Baseball Reference)에 비해 현저히 부족합니다. MLB에서는 FIP, wOBA, WAR 같은 세이버메트릭스가 선수 평가·트레이드·연봉 협상의 기본 언어가 되었지만, KBO에서는 여전히 타율·홈런·타점이라는 전통 지표가 지배적입니다. 이 격차를 직접 해결하기 위해, 데이터 수집부터 분석·시각화·스토리텔링까지 전 과정을 통합한 플랫폼을 구축하기로 결정했습니다.

## 4. 해결 전략

### 전략 1: 7개 크롤러 기반 자동화된 데이터 수집 파이프라인

**문제**
KBO 데이터는 공식 API가 없고, 웹 사이트가 ASP.NET WebForms 기반으로 일반적인 HTTP 스크래핑이 불가능했습니다. 특히 Basic2 페이지는 JavaScript 리다이렉트, PostBack 상태 관리, Double-encoded JSON 등 복합적인 기술적 장벽이 존재했습니다.

**방법론**
기술적 특성에 따라 크롤러를 분화하여, 각각의 장벽에 최적화된 수집 전략을 적용했습니다.

| 크롤러 | 수집 대상 | 기술 | 핵심 해결 이슈 |
|--------|----------|------|--------------|
| KboPlayerStatsCrawler | 선수 통계 (타자 3페이지, 투수 4페이지) | Playwright | ASP.NET PostBack 누적 손상 → 팀마다 URL 재네비게이션 |
| KboTeamStatsCrawler | 팀 순위·기록 (5개 카테고리) | Playwright | IP 분수 변환 ("1279⅓" → 1279.333) |
| KboScheduleCrawler | 경기 일정·결과 | HTTP POST API | Double-encoded JSON 파싱, gameId 역추출 |
| KboGameCenterCrawler | 경기 상세 기록 | HTTP POST API | 스코어보드 + 박스스코어 + 키플레이어 통합 |
| KboPlayerProfileCrawler | 선수 프로필 | Playwright | 2단계 수집 (기록→프로필 연결) |
| KboCrowdCrawler | 관중 현황 | Playwright | 월별 순회 (3~11월) |
| KboFuturesCrawler | 퓨처스 리그 | Playwright | 1군과 동일 패턴 확장 |

**핵심 결과**

| 지표 | 수치 |
|------|------|
| 수집 시즌 | 2020~2025 (6시즌) |
| 총 데이터 건수 | 약 26만 건 |
| MySQL 테이블 | 41개 |
| Flyway 마이그레이션 | 14건 |
| 선수 수 | 1,000+명 |

**인과적 인사이트**
KBO Basic2 크롤링에서 발견한 핵심 교훈은 "워밍업 PostBack"의 필요성입니다. ASP.NET WebForms는 초기 로드 직후 바로 시즌 변경 PostBack을 보내면 실패합니다. 무변경 PostBack을 선행하고, 시리즈를 1(시범경기)→0(정규시즌)으로 먼저 변경해야 정상 동작합니다. 이는 문서화되지 않은 서버 측 상태 관리의 특성으로, 단순 재시도로는 해결할 수 없는 구조적 이슈였습니다.

### 전략 2: 세이버메트릭스 자동 산출 시스템

**문제**
KBO에서 제공하는 기본 지표(타율, ERA)는 수비·행운·구장 효과가 혼재되어 선수의 "통제 가능한 실력"을 분리할 수 없었습니다.

**방법론**
순수 Java 정적 유틸리티 클래스(SabermetricsCalculator)로 11개 파생 지표를 자동 산출하고, 배치 서비스(SabermetricsBatchService)로 시즌별 전체 선수를 일괄 계산하는 시스템을 구축했습니다.

**타자 7개 지표**

| 지표 | 산출 공식 | 분석 의미 |
|------|----------|---------|
| BABIP | (H-HR)/(AB-SO-HR+SF) | 인플레이 안타 비율 → 행운 보정 |
| ISO | SLG - AVG | 순수 장타력 (단타 제외) |
| K% | SO / PA × 100 | 삼진율 → 타격 접근 품질 |
| BB% | BB / PA × 100 | 볼넷율 → 선구안 |
| wOBA | 가중 출루율 (FanGraphs 가중치) | 종합 타격 기여도 |
| AB/HR | AB / HR | 홈런 빈도 |
| PA/K | PA / SO | 삼진 회피 능력 |

**투수 4개 지표**

| 지표 | 산출 공식 | 분석 의미 |
|------|----------|---------|
| FIP | ((13×HR+3×(BB+HBP)-2×SO)/IP)+3.2 | 수비 무관 투구력 |
| K/9 | (SO/IP) × 9 | 탈삼진 능력 |
| BB/9 | (BB/IP) × 9 | 제구력 |
| LOB% | (H+BB+HBP-R)/(H+BB+HBP-1.4×HR) | 위기 관리 능력 |

**핵심 결과**
- ERA vs FIP 괴리 분석으로 "행운의 투수" vs "불운의 투수" 식별 가능
- BABIP 정상 범위(.290~.310)를 벗어난 선수의 다음 시즌 회귀 예측
- wOBA 기반 종합 타격 가치 비교로 전통 타율의 한계 보완

### 전략 3: 5가지 고급 분석 — 순수 Java 통계 알고리즘

**문제**
기본 세이버메트릭스만으로는 "누가 클러치 상황에 강한가?", "이 선수와 유사한 선수는?", "내년 성적은?" 같은 고차원 질문에 답할 수 없었습니다.

**방법론**
외부 ML 라이브러리 없이 순수 Java로 5가지 통계 분석 알고리즘을 직접 구현했습니다.

#### 1) 클러치 지수 — Z-score 가중 합산

```
ClutchIndex = 0.5 × Z(RBI) + 0.3 × Z(HR) + 0.2 × Z(OPS)
Z(x) = (x - μ) / σ
```
- RBI(결정적 타점), HR(결정적 홈런), OPS(전반적 타격력)의 가중 Z-score
- 백분위 랭킹으로 "상위 10% 클러치 히터" 식별

#### 2) 선수 유사도 — 코사인 유사도 (8차원)

```
similarity(A, B) = (A·B) / (||A|| × ||B||)
벡터 차원: [wOBA, AVG, OPS, HR, SB, K%, BB%, ISO]
```
- Z-score 정규화 후 코사인 유사도 계산
- 활용: 트레이드/FA 대체 선수 탐색, 유사 타입 선수 비교

#### 3) Marcel 성적 예측 — 3년 가중평균 + 리그 회귀 + 노화 곡선

```
Marcel = (Y-1×5 + Y-2×4 + Y-3×3) / 12
Regressed = (Marcel×PA + LeagueAvg×1200) / (PA + 1200)
Aging = Regressed × (1.0 - 0.006 × |age - peak|)
```
- 피크 연령: 타자 27세, 투수 28세
- Rate stat(AVG, ERA)만 리그 평균 회귀 적용
- 신뢰도: min(1.0, 보유시즌수 / 3)

#### 4) 승리 예측 — 순수 Java 로지스틱 회귀

```
Features = [ΔERA, ΔOPS, ΔHR, ΔWHIP, HomeAdvantage]
P(HomeWin) = sigmoid(W·X + b)
SGD: Learning Rate=0.01, Epochs=1,000
```
- 2020~(N-1)시즌 학습 → N시즌 예측
- 예측값 범위 제한: [0.25, 0.75] (야구의 불확실성 반영)

#### 5) 피로도 분석 — 등판 간격별 ERA + 전후반기 비교

```
FatigueIndex = ERA(후반기) / ERA(전반기)
등판 간격 그룹: 0~2일(연투) / 3~4일(정상) / 5일+(충분한 휴식)
```
- 올스타 브레이크(7/15) 기준 전후반기 분리
- FatigueIndex > 1.3이면 피로 누적 판정

### 전략 4: Python 배치 분석 + AI 기사 자동 생성

**문제**
세이버메트릭스와 고급 분석의 결과가 수치와 차트에 머물면, 비전문가는 접근하기 어렵습니다. 데이터 인사이트를 "소비 가능한 콘텐츠"로 변환하는 자동화 파이프라인이 필요했습니다.

**방법론: 2단계 파이프라인**

```
[1단계] Python 배치 엔진 ──통계 분석 + 가설검정──▶ DB 저장
                                                        │
[2단계] Java (AiColumnGeneratorService) ──데이터 로드──▶ Gemini API ──기사 생성──▶ 발행
```

**1단계: Python 배치 분석 (15개 주제 + 특집)**

| 주제 | 분석 내용 |
|------|----------|
| 0. 세이버메트릭스 트렌드 | 시즌별 리그 지표 변화 추이 |
| 1. WAR 스포트라이트 | 가치 상위 선수 심층 분석 |

|------|------|

AI에게 "분석해줘"가 아니라 "이 데이터를 기사로 써줘"라고 요청하는 것이 핵심입니다. 분석 로직은 Python과 Java가 전담하고, AI는 이미 검증된 수치를 자연어로 변환하는 역할만 수행합니다. 이렇게 함으로써 환각(hallucination)으로 인한 수치 날조를 원천 차단했습니다.

## 8. 배운 점

### 데이터 수집의 현실: 크롤링은 "기술"이 아니라 "탐정 작업"이다
ASP.NET PostBack 누적 손상, Double-encoded JSON, JS 리다이렉트 등 공식 문서에 없는 장벽들은 Chrome DevTools로 네트워크 요청을 하나씩 추적하며 해결해야 했습니다. 7개 크롤러를 개발하면서 "데이터 수집 비용은 항상 과소평가된다"는 것을 체감했고, 이 경험이 Olist 프로젝트에서 전처리 파이프라인을 설계할 때 큰 자산이 되었습니다.

### 전통 지표의 한계: 숫자 뒤에 숨은 맥락을 읽는 힘
타율 3할 타자가 BABIP .380이라면 그 성적은 행운의 산물일 수 있고, ERA 2점대 투수가 FIP 4점대라면 수비의 도움을 받고 있는 것입니다. 세이버메트릭스를 직접 구현하면서 "결과 지표와 과정 지표를 분리하는 것"이 데이터 분석의 핵심이라는 것을 깨달았습니다. 이 관점은 이후 Olist 프로젝트에서 "평균 지연일수(결과)보다 지연 비율(과정)이 5배 더 강한 예측력을 가진다"는 발견으로 이어졌습니다.

### 분석의 완결: 수치를 넘어 스토리를 만들어야 한다
아무리 정교한 분석도 소비되지 않으면 가치가 없습니다. Python 배치 엔진이 통계적 근거를 생산하고, Gemini AI가 이를 기사로 변환하는 파이프라인을 구축하면서, "데이터 분석가의 역할은 숫자를 만드는 것이 아니라 의사결정을 돕는 이야기를 만드는 것"이라는 확신을 갖게 되었습니다.

### 한계 및 향후 개선 방향
- **실시간 업데이트**: 현재 배치 기반 크롤링 → 실시간 데이터 수집 및 지표 갱신 체계로 전환
- **고도화된 예측**: Marcel 예측의 단순 가중평균 → 베이지안 추론 기반 불확실성 정량화
- **외부 데이터 통합**: 날씨·구장(파크팩터)·관중 등 환경 변수를 분석에 반영
- **배포**: 현재 로컬 개발 환경 → 클라우드 배포로 실사용자 피드백 수집
